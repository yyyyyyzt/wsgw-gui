# 开发指南

## 项目概述

这是一个面向办公电脑的轻量自动化客户端，基于 **Tauri + Midscene.js** 搭建。

## 编码规范

1. **最小可行实现优先**：优先完成可运行闭环，再做增强；禁止一次性堆叠过多未验证功能。
2. **职责清晰**：按“界面层 / 任务编排层 / 平台能力层”拆分文件，避免单文件混杂 UI、业务与系统调用。
3. **命名可读**：函数名使用动词短语（如 `detectChromeDebugPort`、`runBuiltinNewsTask`），状态枚举语义化。
4. **错误可定位**：所有失败路径必须返回可读错误信息，并在日志中保留上下文（模块、操作、失败原因）。
5. **配置外置**：所有可变参数（端口、地址、开关）通过环境变量或配置文件注入，禁止硬编码。
6. **接口先行**：新增模块时先定义输入/输出类型，再实现逻辑，减少跨模块耦合。
7. **文档同步**：任何命令、目录、配置项变化必须同步更新 `README.md` 与 `progress.md`。

## 开发流程规范

1. **任务对齐**：开发前先确认用户的需求，然后在`progress.md`中更新当前任务的状态
2. **最小修改**：只修改需要改动的文件，不要碰无关的底层harness文件
3. **本地验证**：开发完成后，必须先本地运行测试，确保功能正常，不破坏现有功能
4. **进度同步**：开发完成后，更新`progress.md`，标记任务为已完成，记录更新内容
5. **用户确认**：完成后告知用户，让用户验证功能，再进行下一个任务

## 文档协同要求（边开发边补文档）

1. 每完成一个里程碑子任务，必须同步更新 `progress.md` 的对应状态与“文档补齐记录”。
2. 新增脚本命令时，必须在 `README.md` 增加“用途 + 执行命令 + 预期结果”。
3. 新增环境变量时，必须更新 `.env.example`（若存在）和 `README.md` 配置说明。
4. PR 提交前必须执行一次文档一致性检查：命令是否可运行、路径是否存在、步骤是否可复现。

## 提交前检查清单

- [ ] 代码可运行，关键路径已验证
- [ ] `progress.md` 的“当前迭代任务”和“当前状态”已更新
- [ ] `README.md` 的命令与实际脚本一致
- [ ] 未引入违反安全&合规规范的实现

## 安全&合规规范（国企内网强制要求）

1. 禁止添加任何用户数据收集、遥测、外网上报的代码，所有数据必须留在用户本地/内网
2. 禁止硬编码任何敏感信息，所有API密钥、地址必须通过环境变量配置
3. 所有的网络请求，必须是用户明确配置的，禁止偷偷请求外网地址
4. 不能读取用户的本地文件，除非是用户明确授权的配置文件
5. 所有的自动化操作，必须是用户主动触发的，禁止后台自动执行
